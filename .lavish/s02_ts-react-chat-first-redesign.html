transcript·lab goes React — a chat-first TypeScript frontend

Replace the 850-line vanilla-JS src/api/static/index.html with a Vite + React 19 + TypeScript app: land in Chat and just ask, explore the corpus as an interactive Library tree down to individual chunks with a BM25-vs-semantic ranking lab, and give the Scoreboard real model-version and scoring provenance.

00

The verdict, up front

stack
Not TypeScript, not React today.

The whole frontend is one vanilla-JS file with string-concat rendering and zero build tooling. Migration is a real port, but the API layer is clean and stays as-is.

chat
Landing flips from eval-grid to conversation.

One primary answer per question, streamed into a thread. Comparison and judging stay one click away instead of being the default cost of every question.

library
Tree to the chunk level + Retrieval Lab.

All videos → channel → video → chunks, fully interactive. BM25 does not exist in the backend yet — the lab needs one new module and one new endpoint.

scoreboard
Model identity is currently invisible.

ChatAnswer stores no answering-model name. Adding 3 fields at capture time unlocks per-model segmentation, provenance, and honest historical comparisons.

  • 2Agentic by default (D2) — the primary answer comes from rag_agent. While it researches, its per-iteration retrieval lines stream into the bubble (query → chunk count, exactly what the CLI already prints), then collapse to the one-line trace shown. The header compresses agent, latency, tokens, retrieval count, and the RAGAS composite; the composer chip switches to a faster setup any time.
  • 3Progressive disclosure — Compare, Sources (references with timestamp links), and re-judge live on the message, not in the page chrome.
  • 4Three agents, one bubble. When several setups answer the same question (multi-select in the composer, or "Compare setups" on a message), the thread gets one grouped bubble, not three stacked ones: tabs switch between each agent's full answer with its own citations and RAGAS score, the best composite is badged TOP, and the compare grid below lays them side by side. While running, each tab shows a live timer as its agent streams in — the first to finish becomes the active tab, so the fast single-hop answer is readable while rag_agent is still working.
  • 5Composer chips replace the control row: scope (corpus or one video) and default setup are chips; everything else moves behind ⚙ advanced.
  • 6Advanced popover: top_k, auto-judge toggle, multi-setup selection — full workbench power preserved, no longer the landing experience.
Streaming stays. The existing /api/ask SSE events (progress → answer → done) map directly onto the thread: the assistant bubble mounts on progress, fills on answer, and the history rail updates on done. Esc-to-cancel and the live-region announcements are ported as-is.

08

Risks & failure modes

Build step enters a build-free repo. Today uv sync + serve is the whole story; after this, UI changes need Node. mitigation Resolved by D5 — frontend/dist is gitignored, readme documents the Node workflow, and serve fails with a helpful message when the bundle is missing.
Answer renderer drift. The citation/section renderer would exist twice: the TS port (React app) and the Python constant still used by the static chat.html viewer. mitigation Shared fixture tests: the same input snapshots must render equivalently in both, or chat.html generation is retired once the React app covers browsing history.
SSE through the Vite dev proxy. Streaming responses can be buffered by misconfigured proxies, breaking live progress in dev only. mitigation Vite's http-proxy passes streams through; keep X-Accel-Buffering: no (already set in main.py) and verify in P1 before building on it.
History schema growth. New fields on ChatAnswer must not break old entries or the CLI chat path that shares the same file. mitigation All new fields default to None; a round-trip test loads a pre-s02 history fixture verbatim. No migration script needed.
rag_agent latency in a chat UI — accepted per D2. The default answer takes ~30s by design. mitigation Per-iteration retrieval lines stream into the bubble (P2) so the wait reads as active research; Esc cancels; the composer chip drops to the ~6s single-hop setup when speed matters.
BM25 scale ceiling. The in-memory index is rebuilt from all chunk texts; trivial at ~400 chunks, noticeable at ~50k. mitigation Cache the index keyed by collection count; document the ceiling. This is a demo-scale tool, not a search engine.
