02b Zoom in: does this repo get RAG to 100%?

"RAG" in a JD is really ~12 sub-skills. Here's the honest split after the roadmap. Verdict: ~95% of interview-probed RAG competence — and 100% of what a single project can credibly prove. The residual is variant breadth, not depth, and belongs in other repos or a talk-track.

Covered by this repo after P1 + P2

  • Ingestion & normalisation — real API source, async jobs, caching, bulk runs. exists
  • Chunking strategy — segment-aware chunking plus a measured size ablation. exists → P2 makes it defensible
  • Embeddings & vector store — sentence-transformers + Chroma, three collections, metadata filtering. exists
  • Hybrid retrieval — dense + BM25 with RRF fusion. P2
  • Reranking — cross-encoder, 50→10. P2
  • Multi-stage & multi-hop retrieval — summary-first filtering, recursive fan-out, agentic ReAct loop. exists — rare in portfolios
  • Grounding & citations — timestamped source links in every answer. exists
  • Retrieval evaluation — golden dataset, recall@k, MRR, nDCG, CI gate. P1
  • Answer evaluation — LLM-as-judge faithfulness/groundedness. P1
  • Cost/latency trade-off analysis — tokens & latency per strategy, per model. P1 · P4
  • Failure analysis — worked examples of where each retriever loses. P2

The last ~5% — and where it lives

  • Managed / enterprise vector stores (pgvector, Azure AI Search — named in Sirius). Cheapest close: in P3, put storage behind a small interface and add pgvector as a swap-in backend (~3h). This is the one stretch worth doing here.
  • Messy-document ingestion (PDF/OCR/tables — docling territory). Confirmed: a separate project, not this repo — transcripts are clean text and forcing PDFs in muddies the story. Concrete project ideas: (1) extend CUAD-agent with docling PDF parsing — contract clauses are the classic messy-doc corpus and the repo already exists; (2) ASX annual-report Q&A — tables + finance synergy with this repo's corpus; (3) council development-application PDFs — property synergy. Any one of them, run through this repo's P1 harness pattern, completes the pair.
  • GraphRAG / knowledge graphs. Good news: this one can live in this repo and be evaluated against standard RAG — transcripts are entity-rich (people, companies, policies, suburbs, dates). Added as optional Phase 6: extract an entity graph, add a --retriever graph mode, and score it against dense/hybrid through the same P1 harness.
  • Web-scale / multi-tenant retrieval. No side project proves this; it's a talk-track ("here's how I'd shard and isolate tenants"), which the "What I'd do next in production" section (P5) covers.

P1 · Evaluation backbone ★

05 The senior narrative this buys you

Senior isn't "I used LangChain" — it's measured trade-offs, failure analysis, and operational ownership. After the roadmap, your story arc is:

  1. Framing: "I built three retrieval-augmented strategies over the same corpus — single-hop, recursive multi-hop, and an agentic LangGraph loop — specifically so I could measure when agentic complexity pays for itself."
  2. Evaluation: "I labelled a golden dataset and gated CI on recall@k and MRR; LLM-as-judge scores faithfulness. The gate caught a regression when I changed chunking."
  3. Iteration: "Hybrid BM25+dense with a cross-encoder reranker moved recall@10 from X to Y. Here's a query where dense retrieval failed and exactly why."
  4. Trade-offs: "The agent beats single-hop on multi-topic questions but costs ~3× tokens and ~2× latency — here's when I'd ship each."
  5. Operations: "It's containerised, Terraform-deployed to AWS, traced per-request with token cost. I deliberately run a cheap answer model against a stronger judge model — and I can show the cost-per-question trend that justifies it."

06 Risks & how the plan handles them

high

Scope creep. Your ai_plan lists 10+ libraries and 6 project ideas; this repo could try to absorb them all and finish none. Mitigation: this repo is Portfolio Project #2 (RAG + evaluation) only. Text-to-SQL, fine-tuning/LoRA, and memory belong in separate repos — ConvFinQA-agent and data-qa-agent already cover some.

high

Evals skipped under time pressure. The temptation is to jump to hybrid retrieval (more fun). Without P1 first, P2 produces no numbers and the senior story collapses to "I added BM25". Mitigation: phase order is a hard dependency; P2 tasks all consume the P1 harness.

med

Demo corpus optics — resolved (17 Jul). Your call: the corpus stays deliberately mixed (finance/property and AI-engineering), because real users ask a mixed corpus about anything — the agent has to route to the right transcripts. That reframing turns the optics concern into a feature under test: cross-domain routing questions go into the golden dataset and summary-filter routing precision becomes a P1 metric.

med

LLM-judge cost & flakiness in CI. Judge calls are non-deterministic and cost money on every PR. Mitigation: deterministic retrieval metrics gate every PR; judge runs are cached, versioned, and triggered on demand/labels only.

med

Cloud bill drift. A demo service left running quietly costs money. Mitigation: teardown script, scale-to-zero platform choice, billing alarm — and the README documents the one-command local Docker fallback.

07 Decisions — locked 17 Jul 2026

All four came back from your review, plus one scope call. Recorded here with their consequences; the roadmap above already reflects every one of them.

D1 · Eval tooling → RAGAS, with Claude as judge

  • RAGAS supplies the judge-metric vocabulary interviewers recognise: faithfulness, context precision, answer relevance.
  • Your twist, adopted: DeepSeek stays the answering agent; Claude (via claude-agent-sdk) is the judge/teacher — grading answers and flagging weak retrievals. Independent judge ≠ self-grading, which is itself a defensible eval-design choice.
  • Retrieval metrics (recall@k, MRR, routing precision) stay hand-rolled either way — they need your chunk labels, not a framework.

D2 · Cloud → AWS, reusing the data-qa-agent stack

  • Verified reusable from data-qa-agent: Terraform under infra/terraform/, App Runner services, GitHub-OIDC deploy-aws.yml (no stored keys), CloudWatch billing/5xx alarms → SNS, teardown-able by design.
  • This repo copies the pattern with one service (the FastAPI app) — much smaller surface than data-qa-agent's four.
  • Bonus: its Aurora Serverless v2 module makes the P3 pgvector stretch nearly free, closing the enterprise-vector-store gap.

D3 · Observability → Langfuse

  • Named in your highest-fit specs (Sirius, CBA). LangChain callback integration keeps the lift small.
  • Spans for retrieve → rerank → generate; per-request tokens and cost for both DeepSeek answers and Claude judging.
  • MLflow stays for experiment history — the two tools answer different questions and saying why is interview material.

D4 · Corpus → deliberately mixed, routing under test

  • Finance/property stays canonical and the corpus keeps both domains — your framing: real users ask a mixed corpus about anything, so finding the right transcripts is the product.
  • Consequence in P1: golden dataset gains a cross-domain routing question class, and summary-filter routing precision becomes a first-class metric.
  • Scope call, also locked: no LiteLLM router — effort goes to evals + cost tracking of DeepSeek instead; routing stays a documented "next in production" step.
