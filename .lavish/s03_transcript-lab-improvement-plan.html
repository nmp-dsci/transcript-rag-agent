Chat — channel/video scope, visual polish, answer quality

your feedback #1

A1 · Linked Channel ▸ Video scope filters quick win high impact

Today the composer offers Whole corpus or one of 10 videos in a single flat <select>; the API accepts only a video url src/api/main.py:72–80. Your corpus already spans 5 channels, and the Library tree already groups by channel — Chat should too.

Proposed composer — try it, the two filters update each other (live mock, real corpus data)
Scope Agent rag_agent (agentic) ▾ ⚙ advanced
Retrieval scope → whole corpus (10 videos, 281 chunks)
  • Channel picked → video list filters to that channel; retrieval scopes to all its videos.
  • Video picked → channel auto-selects to its owner; retrieval scopes to that one video (current behaviour preserved).
  • Channel cleared → video keeps working standalone; both cleared → whole corpus.
  • Scope chip echoes into the answer bubble's meta chips so judged runs record what they searched.

Why this needs a small backend change: chunk metadata carries only video_id — channel lives solely on the raw_transcripts collection src/rag/storage.py:374. Three ways to bridge it:

Option 1 — Join at query time

Resolve channel → video_ids from raw-transcript metadata (already exposed per video by src/api/corpus.py:55), then call the existing query_by_video_ids() src/rag/storage.py:180. Add channel to AskRequest.

Cost: zero data migration · slightly slower (N per-video queries merged) · channel logic lives in app code
RECOMMENDED

Option 2 — Metadata backfill, no re-embedding

One-off script: Chroma's collection.update(ids, metadatas=…) stamps channel_id/channel_name onto all 281 existing chunks without re-embedding; also add channel to _chunk_metadata() for future ingests. Retrieval then uses a native where={"channel_id": …} filter — one query, clean.

Cost: ~30-line backfill script · runs in seconds at this scale · new ingests get it for free

Option 3 — Full re-index

Re-run index-rag for all 10 videos after extending chunk metadata. Simplest mental model, but re-fetches transcripts and re-embeds 281 chunks for no additional benefit over Option 2.

Cost: minutes of compute + Supadata API calls · risk of transcript drift · unnecessary here

A2 · Visual polish quick win

The Instrument design system (frontend/src/theme.css) is coherent — polish means using data you already store, not a redesign:

  • Video thumbnails — thumbnail_url is already on every raw transcript src/rag/models.py:15. Show them in the scope picker, the Sources list, and the Library tree. Instantly makes the corpus feel real.
  • Suggestion chips grouped by channel — the empty-state prompts are a flat list today; group them under channel microlabels so the two-domain corpus (careers vs property) reads at a glance.
  • Channel color coding — assign each channel a stable hue (like the existing per-setup swatches theme.css:220–222); use it on tree rows, scope chips, source citations, and the future chunk graph.
  • Answer skeleton while streaming — the agent trace streams beautifully, but the final answer pops in as one block; a shimmer placeholder under the trace smooths the ~30s agentic runs.
  • Score strip labels — "FAITHFUL 1.00" with no affordance; add an ⓘ that opens the Workstream-C score breakdown (see C1) so the bubble teaches the metric.

Similarity graph + question overlay

Type a query → its retrieval neighborhood lights up, edges show why neighboring chunks came along. This is the "explain retrieval" view.

Port the PCA scatter as-is

Fastest: reuse projection.json, render an SVG scatter with channel colors and question overlay. No edges — it shows clusters, not relationships. Caveat: PCA explains only 14%+5% of variance here, so distances mislead.

Cost: ~1 day · but limited insight; risks being a pretty dead-end

UMAP layout (better local structure than PCA) with kNN edges drawn on top, retrieval overlay as in option 1. Most informative, adds a umap-learn dependency and layout tuning time.

C

Evals — show how RAGAS judges, then harden it for agents

your feedback #3

C1 · "How was this scored?" — the metrics, made visible

The judge wraps the real RAGAS 0.4.3 metrics src/evals/judge.py:54–114. Internally RAGAS produces per-claim verdicts with reasons, a generated question, and per-chunk verdicts — then the judge keeps only three floats and discards all of it judge.py:116–141. Two deliverables: an explainer (ship immediately) and a per-answer breakdown (needs persistence).

FAITHFUL — is the answer supported?

supported claims ÷ total claims

The judge LLM splits the answer into standalone claims, then NLI-checks each one against the retrieved chunks only.

✓Gold Coast has boom-and-bust history
directly stated in chunk #c2
✓Vacancy swung 0.5% → 9%
figure appears in chunk #c2
✗example: "prices fell 20% in 2024"
not inferable from any retrieved chunk → unfaithful
faithful
1.00

RELEVANT — does it answer the question?

cos( E(genʼd question), E(original) ) × committal

The judge writes the question this answer would answer, embeds both, compares. Evasive answers score 0.

Qasked: "Is the Gold Coast market at risk of collapse, and why?"
Q'generated: "What risks does the Gold Coast property market face?"
similar but not identical → 0.74
relevant
0.74

⚠ Currently computed from a single generated question (strictness=1 judge.py:89) — the noisiest possible setting; see C2.

PRECISION — were the chunks useful, in order?

Σ precision@k × verdict_k ÷ Σ verdict_k

One verdict per retrieved chunk: "was this useful in arriving at the answer?" — rank-weighted, so useful chunks buried low cost you.

✓#1 · #c2 vacancy swings used → P@1 = 1.00
✓#2 · #c7 off-the-plan risk used → P@2 = 1.00
✗#3 · #c11 resume formatting irrelevant — cross-domain bleed
✓#4 · #c1 spruiker marketing used → P@4 = 0.75
precision
0.71

Risks & constraints
  • DeepSeek rejects n>1 — multi-sample judging must loop independent calls (slower, ~3× judge cost) rather than batch.
  • RAGAS internals coupling — capturing intermediates leans on ragas 0.4.3 prompt/callback structure; pin the version and add a contract test so upgrades fail loudly.
  • Judge latency — observed ~57s per 3-answer entry; breakdown persistence doesn't add calls, but golden-set metrics and k=3 sampling do. Budget it in the UI (progress per metric).
  • Metadata backfill is one-way — trivial at 281 chunks, but land the _chunk_metadata() change in the same PR so new ingests never regress.
  • Two-domain corpus — careers vs property makes whole-corpus retrieval look worse than it is (cross-domain bleed shows up in precision); channel scoping in A1 largely defuses this.
