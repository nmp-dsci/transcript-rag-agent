The verdict, up front

Current app, rated against your target workflow: 3.5 / 10. It is a competent chat viewer with a send box — but your workflow is evaluation, and the two journeys that matter most are the two weakest: comparing retrieval methods means scrolling three long bubbles stacked in one column (4/10), and scoring answers doesn't exist at all — no UI, no API, no storage (0/10). The corpus is also invisible: you can't see what's indexed, so you can't know what to ask (1/10).

The good news: the backend already stores almost everything the redesign needs (per-answer metadata, citations, timings, and rich per-video metadata in Chroma — title, channel, duration, upload date, summary). The redesign below reframes the app as an evaluation workbench — three views (Ask & Compare · Library · Scoreboard), side-by-side answer columns, and an inline scoring rubric whose labels become seed data for the roadmap's P1 golden dataset. With the backend additions in §8 (~1 day) plus the rebuilt frontend, every journey in §3 reaches its 10/10 definition: 10 / 10.

Three decisions are yours — styling direction (§6), scoring model (§7), and build scope (§9). Each has a form; queue your picks and press Send to Agent, and I build it.

03 Usability scorecard — the seven journeys

Each journey rated for today's app, with what 10/10 means. Weighted to your stated goal (J3 + J4 are the point of the app), today's overall is 3.5/10.

J1 · Know what you can ask

1/10

The corpus is invisible. You ask blind and hope a transcript covers it.

10/10 = Library view: every indexed video with title, channel, duration, summary; click → “ask about this”.

J2 · Ask a question

6/10

Composer + pills + Enter-to-send work fine. No scope picker, no top_k, no cancel, placeholder teaches nothing.

10/10 = guided scope (corpus/video), advanced params, cancel, suggested questions from summaries.

J3 · Compare retrieval methods

4/10

Three long answers stacked in one column; metadata scattered across chips. Comparison = memory test.

10/10 = synchronized side-by-side columns, aligned metadata strip, per-column timers while running.

J4 · Score & evaluate answers

0/10

Does not exist — no rubric, no winner pick, no persistence. Your end goal has no UI at all.

10/10 = inline rubric per answer + winner pick + note, autosaved, feeding a Scoreboard and exportable labels.

J5 · Revisit past Q&A

5/10

Sidebar list + click works. No search, no filters, no “scored?” marker, no delete.

10/10 = searchable, filterable history with scored/unscored state visible at a glance.

J6 · Grow the corpus

3/10

Modal works for one video; channel bulk looks frozen for minutes and reports only an exit code.

10/10 = live per-candidate log (fetched / cached / failed), history of ingestion runs.

J7 · Trust the system

3/10

One health line. No corpus stats, no visible error states beyond a red bubble.

10/10 = status strip (videos · chunks · stack state), graceful error + retry affordances.

04 Redesign concept — the evaluation workbench

One reframe drives everything: this app is not a chat, it's a lab bench for judging retrieval strategies. Three views, one loop. The loop is the product: every scored answer makes the Scoreboard smarter and seeds the roadmap's P1 golden dataset.

Reading the loop: the Library's “ask about this video” pre-fills the composer → each finished answer lands in its scoring strip → scores aggregate per retrieval method on the Scoreboard → its weak spots suggest the next questions, and every label exports toward the roadmap-P1 golden dataset.

05 Screens & every user journey

Three views cover all seven journeys. What each screen contains, and which journey it closes:

Ask & Compare J2 · J3 · J4 · J5

  • Composer on top — question, scope (whole corpus / pick a video from Library), setup toggles, advanced row (top_k). Enter sends; Esc cancels. J2
  • Answer columns — one per selected setup, side by side with synchronized scrolling; while running each column shows a live elapsed timer and progress state; stacked layout stays available as a toggle for single-setup asks. J3
  • Aligned metadata strip — latency · tokens · chunks · LLM calls in the same row position for every column, so trade-offs read across. J3
  • Scoring strip per column — the rubric you pick in §7, plus a winner pick and optional note; autosaves on click. J4
  • History rail — searchable, filter chips (scored / unscored, setup, video-scoped), scored entries badged. J5

Library J1 · J6 · J7

  • Corpus grid — every indexed video: title, channel, duration, upload date, chunk count, expandable summary (all already in Chroma). J1
  • “Ask about this” — one click pre-fills the composer scoped to that video. J1→J2
  • Index new content — video or channel; per-candidate outcome log (phase 2: live streaming; phase 1: results on completion). J6
  • Status strip — n videos · n chunks · retrieval stack state · last ingestion run. J7

Scoreboard J4 payoff

  • Per-method table — avg score per criterion, win rate, median latency, avg tokens, n scored — the “which retrieval method is actually better” answer, from your own labels.
  • Unscored queue — jump straight to answers awaiting judgment.
  • Export labels — download scores as JSONL: the seed of the roadmap P1 golden dataset. Your clicking literally builds the eval set.

Cross-cutting to 10/10: responsive to tablet width (rails collapse), aria-live announcements for streaming answers, visible focus states, full keyboard path (Enter send · Esc cancel · 1–5 to score the focused answer), and empty states that teach (“Library is empty → index your first video”).

07 Scoring model — pick one

How you judge each answer. This defines the data schema, so it's worth locking before labels accumulate.

ModelPer answer you clickTime/answerWhat it buys youWhat it can't tell you
S1 · Quick verdictOne overall 1–5 + optional winner~5sFast coverage, rough rankingWhy a method lost (no per-criterion diagnostics)
S2 · Rubric-lite recommendedAccuracy · Grounding · Completeness, each 1–5, + winner + optional note~20sDiagnoses why methods win/lose; criteria map ~1:1 onto RAGAS (faithfulness ≈ grounding) so your human labels calibrate the P1 LLM-judgeSlower; rubric fatigue on long sessions (mitigated: keyboard 1–5)
S3 · Pairwise onlyPick the winning answer per question~3sClean win-rate leaderboardNo absolute quality signal; can't spot “all three were bad”

Decision 2 · Which scoring model?

S2 is recommended: it's the only one whose labels directly calibrate the roadmap-P1 LLM-judge, and the winner pick gives you S3's leaderboard for free.

The frontend can only be 10/10 if the API stops hiding things. Small, testable additions — same patterns as the existing create_app factory:

GET /api/scoreboardAggregates scores per setup: averages per criterion, win rate, latency/token medians, n scored. Plus JSONL label export.~1hPure read over history — no new storage.
top_k on AskRequestExposes the retrieval-depth knob the runner already supports.~30mPlumb straight to runner.run(top_k=…).
Cancel askClient abort stops the SSE stream between setups.~1hHonest limit: a setup already executing finishes server-side; the stream just stops. Good enough for a local tool.

high

Score schema churn. If the rubric changes after you've labelled 50 answers, the labels fragment. Mitigation: Decision 2 locks the schema first; the stored record carries a rubric_version field from day one.

med

Redesign scope creep. Three views + scoring + corpus could balloon. Mitigation: build order in §9 ships Ask & Compare + scoring first (your core loop), Library second, Scoreboard third; each slice is demoable alone.
