# Spec: S5 Automated Transcript Ingestion And Pipeline Visualisation

Status: draft
Date: 2026-05-17

## Summary

Today the only way to populate the RAG corpus is to run `index-rag` on a single YouTube URL at a time. S5 adds two automated ingestion modes plus richer dashboard surfaces so a reviewer can see what is in the index, how it grew over time, and how questions land in the embedding space relative to chunks.

S5 is one pipeline. The three changes below are implemented **sequentially in order** so that each stage can be reviewed against a working repo:

1. Bulk ingestion (discovery + `bulk-index` + ingestion run records).
2. Dashboard tab refactor (Transcripts / Ingestion Runs tabs wired to existing data + new run records).
3. Chunk Space tab (PCA projection + persisted artifacts + scatter rendering).

A coding agent should land 1 before 2 and 2 before 3. Earlier stages must not assume later-stage code exists.

S5 covers three changes:

1. **Automated ingestion** of YouTube transcripts via two discovery modes:
   - Channel + time window: pull all videos a given channel published between two dates.
   - YouTube search: take the top N recommended results for a query.
2. **Dashboard upgrades** in `dashboard/rag_pipeline.html`:
   - The existing transcripts view becomes the "Transcripts" tab.
   - New "Ingestion Runs" tab showing historical ingestion jobs and per-job outcomes.
   - New "Chunk Space" tab showing a 2-D scatter plot of chunk embeddings with overlaid test questions and their nearest chunks.
3. **Embedding visualisation research**: pick and document a dimensionality reduction approach (PCA baseline, optional UMAP/t-SNE) so chunks and questions can be rendered on the same 2-D axes for retrieval inspection.

S5 must preserve existing `index-rag`, `ask`, `rag-ask`, and S4 transcript filtering behavior.

**Golden test (see Completion Test for the full check):** bulk-index the latest 5 videos from `@aiDotEngineer`, then render `dashboard/rag_pipeline.html` and confirm chunks from those 5 videos are embedded, PCA-projected, and visualised in the Chunk Space tab.

## Goals

- Add a `discover` step that enumerates candidate YouTube URLs from a channel + time window or a search query.
- Add a `bulk-index` CLI command that ingests every discovered URL using the existing `index-rag` pipeline, with idempotent skip-if-indexed behavior.
- Persist a structured record of every ingestion run (manual single-URL, channel, search) so the dashboard can show history.
- Restructure `rag_pipeline.html` into a tabbed dashboard with at least three tabs: Transcripts, Ingestion Runs, Chunk Space.
- Render chunk embeddings as a 2-D scatter plot in the Chunk Space tab using PCA over the stored embeddings.
- Overlay a small set of test questions on the scatter plot and visually highlight each question's top-k nearest chunks.
- Keep the dashboard a single self-contained HTML file with no external network dependencies at view time.

## Non-Goals

- Do not replace Supadata as the transcript provider.
- Do not introduce a backend server, database, or browser build pipeline.
- Do not implement automatic scheduled ingestion (cron). Bulk ingestion is user-triggered.
- Do not stream live updates to the dashboard. The dashboard is regenerated on demand.
- Do not change S4 transcript-summary filtering semantics.
- Do not introduce a second discovery provider. Discovery must go through Supadata so the project keeps one external provider; see Discovery Strategy.
- Do not introduce a second dimension reduction technique. PCA is the only reducer in S5. UMAP/t-SNE/autoencoders are explicitly out of scope.

## Dashboard Style Requirements

The new tabs and panels must match the existing dark dashboard look defined in `src/dashboard/theme.py` (`DARK_DASHBOARD_STYLE`). The current style already provides `.tabs`, `.tab`, `.tab.active`, and `.panel/.panel.active` classes — reuse them rather than introducing parallel styling.

Hard rules for every new table and panel:

- Dark mode only. No light-mode toggle. Use the existing palette (`#0f141b` body, `#151c26` panels, `#202a36` table headers, `#2d3745` borders, `#8cc8ff` links, `#2f81f7` active accent).
- Tables-heavy layout. Every list (transcripts, ingestion runs, candidates, nearest chunks) is rendered as a table with the existing `table/th/td` styling, not as cards.
- Column-name wrapping. Headers must wrap rather than force column widths. Add `th{white-space:normal;word-break:break-word}` if the existing rule does not already permit wrapping, and never set fixed `width` on `th` or `td` except for narrow numeric columns (≤ 80px).
- No column should "drive" the layout. Use `table-layout:auto` with `max-width` on long-text cells (titles, summary, chunk preview) and `overflow-wrap:anywhere` so a long title cannot stretch the table.
- Rates and ratios render as percentages with one decimal place: `78.6%`, not `0.786`. Apply to indexed/discovered ratios, skip rates, failure rates, and similarity scores when shown alongside other rates.
- Raw cosine similarity scores keep four decimal places (`0.8421`) to match S2/S4 reports; only the *aggregate* ratios are shown as `%`.
- Every table that exceeds ~5 rows must expose filter controls above the table. Filters are plain `<input>`/`<select>` elements styled with the existing dark palette and run entirely in inline JS (no frameworks). Use:
  - Free-text filter on the most identifying column (title, query, video_id).
  - Dropdown filter on the highest-cardinality categorical (mode, status, channel, outcome).
  - Optional numeric-min filter where a score column is present.
- Numeric columns are right-aligned via a `.num` class added to those `<td>`s.
- Long-text cells (summary, chunk text) collapse into `<details>` blocks per the existing pattern; the table cell shows the first ~140 characters and the `<details>` reveals the full text.
- Tab switching uses the existing `.tabs/.panel` JS pattern. Do not add a router or hashchange handler unless deep-linking to a tab is later requested.

### 1. Discovery

Discovery must go through **Supadata** so the project keeps a single external provider. See Discovery Strategy below for the confirmed endpoints (`/youtube/channel/videos`, `/youtube/search`, `/metadata`).

Add `src/transcripts/discovery.py` with two discovery functions returning a normalised list of candidate videos:

```python
class DiscoveredVideo(BaseModel):
    video_id: str
    source_url: HttpUrl
    title: str | None = None
    channel_id: str | None = None
    channel_name: str | None = None
    published_at: str | None = None
    duration_seconds: float | None = None


def discover_channel_videos(
    channel: str,
    published_after: date | None = None,
    published_before: date | None = None,
    max_results: int = 50,
) -> list[DiscoveredVideo]: ...


def discover_latest_channel_videos(
    channel: str,
    limit: int = 5,
) -> list[DiscoveredVideo]: ...


def discover_search_results(
    query: str,
    top_n: int = 10,
) -> list[DiscoveredVideo]: ...
```

Rules:

- `channel` accepts either a channel handle (`@aiDotEngineer`), channel ID, or full channel URL (`https://www.youtube.com/@aiDotEngineer`). Normalise inside the function.
- `discover_latest_channel_videos` is the convenience path used by the completion test; it is a thin wrapper around `discover_channel_videos` sorted by `published_at` desc.
- The time window is inclusive on both ends when both bounds are provided. Either bound may be `None` (open-ended).
- `max_results` and `top_n` cap the candidate list.
- Discovery must not fetch transcripts. It only returns identifiers and metadata.
- Discovery uses `SUPADATA_API_KEY` (already in env). No new credential is added.
- Honour Supadata rate limits with a small backoff identical to the existing transcript fetcher.
- Fail clearly with provider name and reason when discovery cannot complete.

### 2. Bulk ingestion command

Add `bulk-index` to `src/cli.py`:

```bash
uv run python -m src.cli bulk-index channel \
  --channel "@somechannel" \
  --since 2026-01-01 \
  --until 2026-05-17 \
  --max-results 50

uv run python -m src.cli bulk-index channel \
  --channel "@somechannel" \
  --latest 5

uv run python -m src.cli bulk-index search \
  --query "australian capital gains tax reform" \
  --top-n 10
```

Channel mode flags (mutually exclusive groups):

```bash
--channel "@somechannel"   # required: handle, channel ID, or channel URL
--latest N                 # take the N most recently published videos (single Supadata call)
--since 2026-01-01         # inclusive lower bound (requires --until or open-ended upper)
--until 2026-05-17         # inclusive upper bound
--max-results 50           # caps the candidate list when --since/--until is used
```

`--latest` and `--since/--until` are mutually exclusive. Exactly one of them must be supplied.

Search mode flags:

```bash
--query "..."              # required
--top-n 10                 # caps the candidate list
```

Common flags (apply to both modes):

```bash
--dry-run                  # discover only, do not index
--skip-existing            # default true; see Idempotency below
--no-skip-existing         # force re-index even when chunks already exist
--refresh-summary          # forward to index-rag for each URL
--concurrency 1            # keep sequential by default for the prototype
--label "channel-x-2026q1" # optional human label saved on the ingestion run
--no-discovery-cache       # bypass the 24h discovery cache for this run
```

Behavior:

1. Run discovery, log the candidate list.
2. For each candidate, call the existing single-URL indexing path used by `index-rag` so summaries, summary embeddings, and chunks stay consistent with S4.
3. Record per-URL outcome: `indexed`, `skipped_existing`, `failed` with error text.
4. Persist an ingestion run record (see Data) regardless of success.
5. Print a summary table: discovered, indexed, skipped, failed.

Idempotency:

- "Indexed" predicate: a `video_id` counts as indexed only when it is present in **both** `raw_transcripts` **and** `transcript_chunks`. A row present in only one store is treated as incomplete and re-indexed even with `--skip-existing`. This prevents a previous crash between raw-fetch and chunk-write from poisoning the cache.
- `--skip-existing` (default): skips the raw-fetch and chunking stages for already-indexed video IDs. It does **not** skip the summary stage when `--refresh-summary` is also set — those flags compose:
  - `--skip-existing` alone → fully indexed videos are skipped end-to-end.
  - `--skip-existing --refresh-summary` → raw+chunks reused, summary regenerated.
  - `--no-skip-existing` → all stages re-run regardless of cache state.
- Outcome label reflects what actually happened: `skipped_existing` (no work done), `indexed` (raw or chunks written), `summary_refreshed` (chunks reused but summary regenerated), `failed` (with error text).
- Re-running the same channel/window or search must re-discover and re-evaluate every candidate against this predicate, and still write a fresh ingestion run record even when every candidate is skipped.

Error handling:

- A single URL failure must not stop the run. Each failure is captured on the run record and surfaced in the dashboard.
- Discovery failure aborts the run and writes a failed run record with stage `discovery`.

### 5. Embedding visualisation pipeline

#### Background research: dimension reduction options

Goal: project ~hundreds–thousands of 384-dim MiniLM chunk vectors into 2-D for a scatter chart, with a single example question projected into the same axes. The shortlisted techniques and the rationale for the chosen default:

| Technique | Notes | Verdict for S5 |
|---|---|---|
| **PCA** | Linear, deterministic, fast, fits `(n_chunks, d)` in milliseconds, supports out-of-sample `transform` so the question can be projected through the same fitted projector. Captures variance directions; clusters that differ on a non-linear manifold may collapse. | **Chosen.** Matches the prototype's bias for boring, testable, deterministic code. |
| **t-SNE** | Non-linear, preserves local neighbourhoods well. Stochastic; no native out-of-sample transform — the question would need a parametric variant or a nearest-neighbour proxy. Slow on large corpora. | Rejected — out-of-sample is awkward and the layout changes between renders. |
| **UMAP** | Non-linear, fast, supports out-of-sample `transform`. Stochastic by default but seedable. Extra dependency (`umap-learn`, plus `numba`). | Rejected for S5 — keeps dependencies minimal and the projection deterministic across renders. |
| **Truncated SVD** | Linear, equivalent to PCA on centred data, useful for sparse inputs. No advantage over PCA for dense MiniLM vectors. | Rejected. |
| **Autoencoders / parametric projections** | Out of scope for a prototype. | Rejected. |

Decision: **PCA (`sklearn.decomposition.PCA`, `n_components=2`) is the only reducer in S5.** No flag, no alternative, no follow-up reducer in this spec.

#### Module

Add `src/dashboard/chunk_space.py`:

```python
@dataclass(frozen=True)
class ChunkProjection:
    method: str               # always "pca" in S5
    fitted_at: str            # ISO timestamp
    embedding_model: str
    n_chunks: int
    explained_variance: tuple[float, float]
    components: list[list[float]]   # 2 x d, for out-of-sample transform
    mean: list[float]               # d, for centring
    chunk_coords: list[tuple[str, float, float]]  # (chunk_id, x, y)


def fit_chunk_projection(
    chunk_embeddings: np.ndarray,
    chunk_ids: list[str],
    embedding_model: str,
) -> ChunkProjection: ...


def transform_question(
    projection: ChunkProjection,
    question_embedding: np.ndarray,
) -> tuple[float, float]: ...


def nearest_chunks_for_question(
    question_embedding: np.ndarray,
    chunk_embeddings: np.ndarray,
    top_k: int,
) -> list[NearestChunk]: ...
```

Rules:

- Use `sklearn.decomposition.PCA` with `n_components=2`. Add `scikit-learn` to dependencies and document this in `readme.md`.
- Fit PCA on chunk embeddings only, then `transform` the question embedding using the same `components` and `mean` so chunks and the question share an axis system.
- Compute nearest-chunk scores in the **original** embedding space, not the 2-D projection, so the visual is a layout aid and the ranking remains faithful to retrieval.
- Embed the projection JSON and per-question nearest-chunk lists inline in the generated HTML (script tag with `type="application/json"`) so the page is self-contained.

#### Persisted projection artifact

The fitted projection must be saved to a versioned, committable path so anyone cloning the repo can open `rag_pipeline.html` without re-running the corpus:

```text
dashboard/chunk_space/
  projection.json        # ChunkProjection serialised (components, mean, chunk coords, metadata)
  question.json          # example question text, embedding, projected coords, nearest chunks
```

Rules:

- `dashboard/chunk_space/` **is committed to git** so a fresh clone can open `rag_pipeline.html` and see a populated Chunk Space tab without re-running ingestion. Reproducibility-on-clone is the explicit goal; the noisy-diff cost is accepted.
- **The projection is refit at dashboard render time, not during ingestion.** `src.dashboard.rag_pipeline` is the sole writer of `projection.json` and `question.json`. `index-rag` and `bulk-index` never touch these files. Rationale: the dashboard is regenerated on demand and is the natural cache-invalidation point; keeping refit out of the ingestion hot path removes an unrelated failure mode.
- On render, the dashboard generator:
  1. Loads all chunk embeddings from `transcript_chunks`.
  2. Fits PCA (`n_components=2`) and projects every chunk.
  3. Projects the example question through the same `components`/`mean`.
  4. Computes top-k nearest chunks in the **original** embedding space.
  5. Writes `dashboard/chunk_space/projection.json` and `dashboard/chunk_space/question.json`.
  6. Inlines that JSON into the generated HTML.
- A `--refresh-projection` flag forces recomputation. Without it, render reuses cached files when the chunk-count fingerprint stored in `projection.json` matches the live `transcript_chunks` count, otherwise it refits automatically.
- Refit cost is bounded: PCA on MiniLM-dim vectors at prototype corpus sizes is well under one second. If `n_chunks` grows beyond 50,000, fall back to `sklearn.decomposition.IncrementalPCA`; until then, plain `PCA.fit_transform` is fine.
- File size cap: hard-fail the render if `projection.json` would exceed 5 MB. On overflow, leave the previous projection in place and log a clear warning so the dashboard still renders.
- Scatter render cap: when `n_chunks > 5000`, the SVG renders a stratified sample (by `video_id`) of 5000 points. The full chunk count is shown in the tab header. Nearest-chunk highlights for the question are always rendered regardless of the sample.

Test question source:

- A single canonical example question is configured in code (the capital-gains tax question already used in S4) and can be overridden via `--question` on the dashboard CLI.
- The dashboard displays exactly that one question. There is no multi-question overlay and no question selector in S5.

## Acceptance Criteria

- `bulk-index channel` discovers and indexes videos for a given channel using either an explicit time window or `--latest N`, skipping already-indexed video IDs by default.
- `bulk-index search` discovers and indexes the top N search results for a query via Supadata's `/youtube/search` endpoint.
- A failed transcript does not abort the run; the failure is captured in the run record.
- Every bulk run writes one JSON file under `.yt-agent/ingestion_runs/`, including runs where every candidate was skipped (cache-hit case).
- `rag_pipeline.html` exposes three tabs — Transcripts, Ingestion Runs, Chunk Space — using the existing `.tabs/.panel` styling.
- Every table follows the Dashboard Style Requirements: dark palette, wrapping column headers, no width-driving columns, percentages with one decimal, filters above any table with more than ~5 rows.
- Ingestion Runs tab shows all historical runs with per-candidate outcomes, sortable/filterable by mode and status.
- Chunk Space tab shows a 2-D PCA scatter of chunk embeddings with exactly one example question overlaid, its top-k nearest chunks highlighted, and a side table listing those chunks.
- Nearest-chunk scoring is computed in the original embedding space, not the 2-D projection.
- The PCA projection and question result are persisted under `dashboard/chunk_space/` and committed so a fresh clone renders the dashboard without re-running embedding.
- The dashboard generator is the sole writer of `dashboard/chunk_space/projection.json` and `question.json`. `index-rag` and `bulk-index` never touch those files. The dashboard refits automatically when the chunk-count fingerprint stored in `projection.json` no longer matches the live `transcript_chunks` count, and unconditionally when `--refresh-projection` is passed.
- Running the completion test against `@aiDotEngineer` indexes the latest 5 videos on first run and skips all 5 on the second run.
- Existing `index-rag`, `ask`, `rag-ask`, and S4 transcript-summary filtering paths still work and tests still pass.

## Verification

- Tests:
  - `discover_channel_videos` returns normalised candidates with mocked provider.
  - `discover_search_results` returns top-n candidates with mocked provider.
  - `bulk-index` skips already-indexed video IDs unless `--refresh-summary` is set.

## Discovery Strategy (Supadata)

Confirmed against the live Supadata docs (https://docs.supadata.ai/) at S5 spec time. Supadata is the *only* external service the project depends on for both discovery and transcript fetching.

### Confirmed Supadata endpoints

All endpoints sit under `https://api.supadata.ai/v1/` and require an `x-api-key` header. The project already loads `SUPADATA_API_KEY`; no new credential is needed.

**1. Channel videos — `GET /youtube/channel/videos`**

- Query params:
  - `id` (string, required) — accepts channel URL, handle (`@aiDotEngineer`), or channel ID.
  - `limit` (number, optional) — max results.
  - `type` (enum, optional) — `video` | `short` | `live` | `all`. Use `video` for the prototype (no Shorts, no live).
- Response shape:
  ```json
  { "videoIds": ["..."], "shortIds": ["..."], "liveIds": ["..."] }
  ```
- Ordering: "videos are returned latest first" matching YouTube's channel page. **There is no date-range filter and no `publishedAt` is returned per item.**
- Implication: `--latest N` is a single call. `--since/--until` requires a second hop per ID to the metadata endpoint to read `createdAt`, then filter and trim.

**2. YouTube search — `GET /youtube/search`**

- Query params:
  - `query` (string, required) — free-text search.
  - `limit` (number, optional) — results per page.
  - `nextPageToken` (string, optional) — pagination cursor.
- Response: paginated `YoutubeSearch` schema (matching videos, channels, playlists). The exact shape is documented as a schema reference; the client maps the video items into `DiscoveredVideo` and discards channels and playlists.

**3. Per-video metadata — `GET /metadata?url=<videoUrl>`** (used for date-window filtering)

- Query param: `url` (URL-encoded YouTube video URL).
- Response includes `id`, `title`, `author.displayName` (channel name), `additionalData.channel_id`, `media.duration`, `stats.views`, and `createdAt` (ISO 8601). `createdAt` is the upload timestamp and is what S5 stores as `published_at`.
- The legacy `/youtube/video` endpoint is deprecated; use `/metadata`.

### Mapping project verbs to Supadata calls

| Project verb | Supadata call | Notes |
|---|---|---|
| `discover_latest_channel_videos(channel, limit)` | `GET /youtube/channel/videos?id={channel}&limit={limit}&type=video` | Single call. Native latest-first order. No metadata hop required for the completion test. |
| `discover_channel_videos(channel, since, until, max_results)` | `GET /youtube/channel/videos?id={channel}&limit={max_results}&type=video`, then `GET /metadata?url=...` per ID to read `createdAt` for the date filter | Two-stage. Stop hitting `/metadata` as soon as a video's `createdAt` is older than `since`, since results are latest-first. |
| `discover_search_results(query, top_n)` | `GET /youtube/search?query={query}&limit={top_n}` | Map video items to `DiscoveredVideo`; ignore channel and playlist results. |
| Channel handle/URL → canonical form | Pass the user input through to `/youtube/channel/videos`; the `id` param accepts URL, handle, or channel ID natively. | No separate lookup call needed. |

### Implementation steps

1. Add a thin Supadata discovery client under `src/transcripts/discovery.py` reusing the existing Supadata HTTP session and timeouts.
2. Map endpoint responses into `DiscoveredVideo`, dropping entries that lack a `video_id`. `published_at` is populated from `/metadata.createdAt` when the metadata hop runs, and `None` otherwise (e.g. for `--latest N` channel discovery, where the date is not needed).
3. Translate Supadata pagination (`nextPageToken`) into a single bounded loop capped by `top_n` / `max_results` for the search path.
4. Cache the discovery response per `(mode, channel|query, params, day)` under `.yt-agent/discovery_cache/` so repeated dry-runs don't burn quota. Cache TTL: 24 hours, configurable via `YT_AGENT_DISCOVERY_CACHE_TTL_HOURS`. Also cache `/metadata` responses keyed by `video_id` since they are stable.
5. Tests stub the Supadata client at the HTTP layer.

### Known limitations carried into S5

- **Date-window discovery is a per-video metadata fan-out**, not a server-side filter. Acceptable for prototype corpus sizes; the cache plus latest-first ordering bounds the call count.
- **Channel-videos response carries no titles or channel name**, only IDs. The bulk indexer's per-candidate row in the ingestion run record will populate `title` and `channel_name` from either `/metadata` (date-window mode, already hopped) or from the transcript fetch result (latest mode, populated post-index). Until populated, the ingestion run record stores `null` for those fields and the dashboard renders them as `—`.
